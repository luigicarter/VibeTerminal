# Frontend

The `frontend/` folder contains the Vite React renderer that users see inside the Electron window.

## Files

- `frontend/main.tsx` - React entry point mounted by `index.html`.
- `frontend/App.tsx` - Top-level workspace UI, localStorage persistence, project selection, code-change line totals, session lifecycle, and agent launcher controls. On app reopen, `restoreSession` brings every pane back as a **fresh** terminal (never an auto-resumed chat): a threaded pane's prior thread is moved into `resumeRef` and a new chat is launched, so "restore my workspace" is decoupled from "resume my conversation". `resumeSession` is the explicit opt-in that relaunches the stashed `resumeRef` thread — or, when a chat pane's resume picker (Open Fusion AND Fusion) passes a specific `targetRef`, that saved chat instead (the outgoing conversation becomes the next `resumeRef` either way, and a thread already claimed by another pane is refused).
- `frontend/sessionLaunch.ts` - Builds launch/resume commands for Codex, Claude, OpenCode, Cursor, and plain terminal sessions. Claude launches pin `--session-id` but deliberately never pass `--name`: forcing the pane's placeholder label ("Claude 2") onto the session overrides the title Claude generates from the first prompt in its own `/resume` picker. `createThreadRef` mints refs without a title for the same reason — `threadRef.title` only ever holds the provider's own generated/custom title, harvested back from local metadata.
- `frontend/types.ts` - Shared renderer-side domain types for agents, sessions, layouts, thread lookup results, and terminal events.
- `frontend/cwdConflicts.ts` - Pure helpers detecting agent panes that share (or nest inside) one working folder across every scope (Multi + all workspaces), plus the chip label/tooltip formatting. Normalization mirrors `normalizeWorkspacePath` (unconditional lowercase, no symlink resolution — advisory-only impact).
- `frontend/electron.d.ts` - Type declarations for the `window.vibe` preload API used by the renderer.
- `frontend/vite-env.d.ts` - Vite environment declarations.
- `frontend/styles.css` - App shell, sidebar, toolbar, terminal pane, status, and tiled-board styling.

## Components

- `frontend/components/TerminalPane.tsx` - Owns the xterm instance for a session (pinned to the session id so it survives launch-command changes), sends terminal input/resize IPC calls, receives PTY events, relaunches the PTY only on a launchToken bump, and polls for agent thread metadata. Renders a "Resume last chat" control (wired to `onResume`) only when the pane has a `resumeRef`. The header shows `threadRef.title || session.name`: for claude/codex panes with a thread id but no title yet, a bounded-backoff title harvest polls the `confirmId` lookup (which returns a titled ref) and an Enter keystroke restarts the chain — so the placeholder "Claude 2" is replaced by the provider-generated title once the first prompt exists.
- `frontend/components/ocChat.tsx` - The shared OpenCode-TUI-parity chat kit both chat panes render through (extracted from OpenFusionChatPane 2026-07-03 so the Fusion pane could wear the same skin). Purely presentational — it knows nothing about either pane's event stream: the `OcChatMessage` row model, `OcChatRow` (user blocks, markdown prose for the pane's `proseRole`, one tool row per callID with OpenCode's icon glyphs/states, bash block panels, Edit diff panels, `todowrite` checklists, Task delegation rows with role chips and "↳ …" progress lines, "+ Thought" collapsibles, subagent "↳" worklines, "▣ … · duration" turn-end lines, error panels), plus `OcSpinner`, `Markdown` (react-markdown + GFM, links open externally), `DiffBlock`, the block-letter `OcLogo` wordmarks (`OC_LOGO_OPEN`/`OC_LOGO_FUSION`), and the tool glyph/label helpers (which understand both OpenCode's `filePath` and Claude Code's `file_path` argument spellings). The visual skin is the `.oc-skin` CSS scope in styles.css — both panes put it on their root article; `.openfusion-pane` keeps only OF-specific extras (role chip, auth rows, gate, steering colors).
- `frontend/components/FusionChatPane.tsx` - The Fusion (Claude + Codex) chat pane: normalizes the `fusion-chat:event` stream into shared `OcChatRow` rows (proseRole "opus"), so it renders with the same OpenCode-TUI parity as the Open Fusion pane (2026-07-03 reskin — the old bubble-row transcript, control strip, rounded composer, and send button are gone). Tool calls are ONE row per toolId — created on `tool-call`, settled in place by `tool-result` (done, or red via the host-forwarded `isError`; still-running rows settle as "aborted" on interrupt/error/close). Claude tool names map to the OpenCode vocabulary (`Read`→`read`, `TodoWrite`→`todowrite`, …); Edit rows derive their colored diff panel from the input's `old_string`/`new_string` (capped at 80 lines — Fusion has no host metadata channel like OpenCode's `meta.diff`); `codex_investigate`/`codex_implement` render as OpenCode Task rows ("Scout/Executor Task — task") whose live "↳ …" line is ticked by the Codex side-channel's command/file/message activity (each tick also lands as a Details-lane workline), completed as "↳ N updates · 12s", click revealing a report composed from the bridge's JSON (findings/summary, files, verifier verdict); the other bridge calls (`codex_goal_*`, `codex_respond`, `codex_cancel`, ExitPlanMode) are one-liner tool rows with friendly titles. Opus prose is markdown; thinking is the spinner → "+ Thought" collapsible; every turn closes with "▣ Fusion · model · duration" ("· interrupted" on aborts — the separate interrupt notice rows are gone). Empty state = the block-letter FUSION wordmark. Composer = OpenCode's prompt box (no send button, Enter sends; the meta row inside holds the clickable Plan/Auto mode chip — Shift+Tab still toggles — plus the planning ⇄ execution model pair) with the status row below (knight-rider spinner + "esc interrupt" while busy; last-turn cost + shift+tab/`/help`/enter hints when idle). `/models` prints the current models, efforts, and per-role fast-serving states; `/fast` toggles real fast serving without changing model/effort, while `/speed quick` is the renamed low-effort/lighter-planner downgrade preset. Footer = OpenCode's status bar: cwd (parent muted, name bright), "△ 1 Decision" while an approval/question is pending, the `/details` toggle (also a slash command; default ON — off hides completed-successful tool rows and internal rows, keeps running/failed/Task rows), and the "• Fusion" brand. Fusion's own panels survive the reskin: the decision approval panel, the plan-accept bar, steering pins, and the settings slash palette. Mid-turn sends (steer) pin above the composer with a QUEUED badge until the next assistant message (`turn-start` = the next API call, whose context includes them) absorbs them into the transcript as `Steer: …`; turn end/interrupt/error/close flushes any still-pinned steer as the freshest entry so it is never buried mid-stream (mirrors the Open Fusion pane's steering pin). Streaming render model (shared with the Open Fusion pane): deltas batch into ONE setMessages per animation frame (flushed synchronously before any non-delta event so ordering never changes), transcript rows are memoized components (a delta re-renders only the growing bubble), and autoscroll is bottom-pinned — it follows the stream only while the user is at the bottom (scrolling up stops the yank; scrolling back or sending re-pins). Per-delta full-transcript re-renders plus unconditional scroll-yanking were the "choppy and slow" streaming UX (2026-07-03). `/resume` and the header's resume button open a saved-chat picker over `window.vibe.agentThreads.list` with `provider: plannerFamily` and `fusion: true` (this folder's Fusion-created chats, newest first, generated titles + ages, type-to-filter — a claude planner lists claude chats, a codex planner codex threads); selecting a row hands its threadRef to `onResume(threadRef)`, rows active in another pane (`claimedThreadIds`) are marked and refused, the current conversation is excluded, and a listing failure offers a "Resume last chat" fallback against the stashed `resumeRef`. The resumed conversation is REHYDRATED into the transcript by the host (replay-tagged user/assistant events plus a `result subtype:"restored"` settle the pane status-neutrally) — a resumed pane shows the prior chat, not a fresh hero.
- `frontend/components/OpenFusionChatPane.tsx` - The Open Fusion chat pane over `window.vibe.openFusionChat` (headless per-pane `opencode serve`; the OpenCode TUI is not rendered — the pane renders OpenCode's TUI row shapes, verified against the v1.17.11 `packages/tui` source, wearing Open Fusion's palette and branding; the row renderer and chrome pieces live in the shared `ocChat.tsx` kit, rendered here with proseRole "brain"). Empty state = a block-letter OPEN FUSION logo in OpenCode's exact glyph style (left word muted, right word bold accent) + OpenCode's "Ask anything..." cycling placeholders. Transcript = OpenCode part renderers: user blocks with an accent left bar on a panel background; Brain markdown prose (react-markdown + GFM; links open externally via `openExternal`); ONE tool row per callID created on `tool-call` and updated in place by its `tool-result` (OpenCode icon glyphs — `$ → ✱ ← % ◈ ⚙ │/✓` — with "~ pending" before input, bright running, muted done, red click-to-expand failed, strikethrough denied); bash as a `$ command` block panel with output collapsed at 10 lines ("Click to expand"); Edit as a colored unified diff from the host-forwarded `meta.diff`; `todowrite` as OpenCode's "# Todos" `[✓]/[•]/[ ]` checklist; thinking as a "Thinking" spinner → "+ Thought: title" collapsible; `task` delegations as OpenCode Task rows (Executor/Investigator chip, "Executor Task — description" + live "↳ current tool" line, "↳ N toolcalls · 12s" when done, click reveals the extracted task report). Parallel executor progress is keyed by child `sessionID`, so two executor children can update different live Task rows at once. Every turn closes with "▣ Brain · model · duration" ("· interrupted" on aborts). The composer is OpenCode's prompt box — accent left bar, agent · model meta row inside, NO send button (Enter sends), and a status row below with the knight-rider block spinner + "esc interrupt" while busy or token·cost + `/help` hints when idle. The pane footer is OpenCode's status bar: cwd (parent muted, name bright), "△ 1 Permission" when pending, the `/details` toggle (default ON like OpenCode; off hides completed-successful tool rows and subagent "↳" worklines, keeps running/failed/Task rows), and the "• OpenFusion" brand. Streams each OpenCode part into its own bubble keyed by the host's `streamId` (concurrent parts — parallel Scouts, reasoning beside text — used to interleave chunk-by-chunk into one garbled paragraph), pins mid-turn sends above the composer with a QUEUED badge until the host's `step-start` signals the Brain's next step absorbed them (flushed into the transcript at that point; on interrupt/result/error they flush as the freshest entry so steering is never lost — and an abort's trailing `result` no longer overwrites the interrupted waiting-state with done), answers OpenCode permission asks in-pane with OpenCode's exact options ("Permission required" — Allow once / Allow always / Reject; Enter allows when the composer is empty, Esc rejects and only interrupts when no ask is pending), and hosts `/brain-model` + `/executor-model` pickers fed by the server's provider catalog — Brain picks apply live, Executor picks restart the pane via `onSettingsChange`. The picker matches opencode's `/models` feel: typing fuzzy-searches models across ALL connected providers in one shot (provider-first browsing stays available with an empty filter), and a successful connect drills straight into that provider's model list for the role that led there (bare `/connect` targets the first unset role). There are NO default models: a first-run gate panel walks connect-a-provider → pick Brain → pick Executor (fed by `GET /config/providers`, requested as soon as the engine reports its session), and non-slash turns are blocked until both roles are set. Provider auth is in-pane with full OpenCode parity: "needs auth" picker entries and `/connect <id>` walk the provider's real auth methods from `GET /provider/auth` — method choice when there are several, prompt fields (text/select), masked API-key entry whose prompt answers become credential `metadata`, or browser OAuth (authorize → open/copy URL → paste code or auto device-flow wait). `/connect` refuses ids that are not in the catalog (a stored key for an unknown id would never be used). `/disconnect <id>` removes a credential. Everything lands in the app-owned OpenCode credential store under vibeTerminal's userData (`openfusion/opencode-home/`), never in the user's personal OpenCode install. Thread confirm-before-resume passes `openFusion: true` so discovery reads the same app-owned store. `/resume` and the header's resume button open a saved-chat picker over `window.vibe.agentThreads.list` (every app-created session for the pane's folder, newest first, titles + ages, type-to-filter): selecting a row hands its threadRef to `onResume(threadRef)`; rows active in another pane are marked and refused, the current conversation is excluded, an empty store reads "No saved chats", and a listing failure reads "Couldn't read saved chats" with a "Resume last chat" fallback against the stashed `resumeRef` — never as an empty history.
- `frontend/components/TiledBoard.tsx` - Drag/resize layout engine for terminal panes, including collision handling, swaps, adjacent resize behavior, and persisted fluid layouts.

## Working / done signals

Two sidebar signals share the dot slot on each folder button and the Multi-mode card (`App.tsx`), with the blue attention dot taking priority over the working spinner:

- **Blue attention dot** (`shouldShowAttentionDot`) - a pane finished a turn / needs you (`attention.unread` + `waiting|completed|failed`). Cleared by selecting the pane.
- **Amber working spinner** (`.attention-dot-working`, gated by `isSessionWorking` = `status === "running"`) - a pane is actively running a turn.

"Working" must mean the agent is working, not the user interacting, so `status === "running"` is set by exactly one source per pane kind — never by raw output that could be a keystroke echo or a focus/click redraw:

- **claude / opencode / cursor** (`isTurnTelemetryKind`) - driven purely by turn telemetry: the `agent-running` event with `turnStart:true` (a genuine turn start) forces `running` via `applyAgentRunning` even past the done/failed latch; `turnStart:false` (claude tool activity) goes through `reconcileStatus`, so a tool hook whose POST races past the turn's `Stop` cannot resurrect a finished pane. `agent-attention` settles status to `waiting`/`done`. `TerminalPane` never marks these working from output, and `applyTerminalStatus` ignores a `running` derived from their terminal events. (cursor's turn start is its `beforeSubmitPrompt` hook; it emits no `waiting`, so `running` simply holds until `stop`.)
- **codex / plain terminals / others** - the `TerminalPane` output heuristic (`markActiveFromOutput`): output reads as "working" unless it lands within `INPUT_GRACE_MS` of the user's last keystroke/click (echo/redraw suppression). The App-level listener no longer derives `running` from `data`, so background non-agent panes show no spinner (no mounted heuristic) — by design. When a mounted pane unmounts while still marked `running` by the heuristic, cleanup settles it back to `waiting` before discarding the mounted idle timer, which prevents folder spinners from getting stuck on "working" after switching workspaces or maximizing another pane. The same cleanup also settles a stale `starting` pill to `waiting` if the pane unmounts during initial boot.

The done/failed pill is a latch (`reconcileStatus`): trailing output — the shell prompt returning after an agent exits, a stray byte — must never flip a finished pane back to working/waiting. What releases or settles a stuck pill is **keyboard input** into the pane (`statusAfterUserInput` in `attention.ts`, applied through the pane's `onInputStatusRelease` callback which bypasses `reconcileStatus`):

- **codex / plain terminals** - typing is their equivalent of claude's `UserPromptSubmit` (codex has no turn-start telemetry, so without this a codex pane latched `done` by its turn-end notify could never show working/waiting again until restart). The latch releases to `waiting`; real output then flips it to `running` via the heuristic.
- **claude (approval waits only)** - answering a permission prompt has no hook of its own (`PreToolUse` fires before the prompt, `PostToolUse` only when the tool ends), so the answer keystroke flips `waiting`->`running` immediately instead of reading "requires feedback" for the whole tool run. Idle/question waits stay put — composing a prompt is not working.
- **telemetry kinds, bare Esc while `running`** - the TUI interrupt key. No hook fires for an interrupt (claude's `Stop` does not), so Esc settles the pill to `waiting` immediately; if it merely dismissed a menu, the next telemetry event re-asserts `running` (waiting->running is never latched).
- `isHumanTerminalInput` filters out terminal-generated reports (focus in/out, mouse reports, arrow keys — anything starting with a bare ESC except a bracketed paste), so clicking or focusing a finished TUI pane never disturbs its pill.

Two more guards keep pills truthful across missed hooks and remounts:

- **Stale-running watchdog** (`TELEMETRY_RUNNING_QUIET_MS`, `armTelemetrySettle` in `TerminalPane`): a telemetry-kind pane that stays `running` with zero PTY output for ~12s settles to `waiting` — these TUIs repaint their spinner constantly while genuinely working, so prolonged total silence means the turn ended without its hook (interrupt, lost POST). The settle fires only if the status is unchanged since arming, so a `starting`->`running` transition never settles a fresh turn on the shorter boot delay.
- **Remounts are replays**: a `snapshot` is buffered old output (workspace switch, maximize), so it never marks a pane working (`statusFromTerminalEvent` returns null for a running snapshot; the pane only re-arms the telemetry settle from it), and the launch effect only sets `starting` when the status is `idle` — every genuine launch path (create/restart/resume/settings change) resets to `idle` first, so a remount of a live pane never disturbs a settled `done`/`failed`/`waiting` pill.

The Fusion / Open Fusion chat panes' header pill takes its live states (working/waiting/failed) from the pane's local flags, but its settled state from the app-reconciled `session.status` — the same source the sidebar reads: `done` after a completed turn, `waiting` after an interrupt (the pane is blocked on the user, not finished), `failed` when the last turn errored. Only a pane with nothing settled reads "ready". Before this, everything non-busy collapsed into "ready", so a pane that was just waiting for input was indistinguishable from one that had finished a task.

Two rules keep the chat panes' `done` from becoming a permanent state (2026-07-04):

- **Chat-pane reattach replays are status/attention-neutral.** When a chat pane remounts onto a live host session (project switch back), the host re-emits its buffered history tagged `replay: true` (`fusionChatHost.cjs` / `openFusionChatHost.cjs` `replaySession`). Replayed events rebuild the pane's transcript and local flags only: the pane's `reportStatus`/`reportAttention` wrappers drop app-level emission for them, both App lifecycle mirrors (`applyFusionChatLifecycle` / `applyOpenFusionChatLifecycle`) skip them outright (they tracked the live events while the pane was unmounted, so they already hold the settled state), and `main.cjs` doesn't re-forward replayed `background-activity`. The panes' launch effect also only emits `starting` when the status is `idle`/`starting` — same rule as `TerminalPane`'s remount guard above. Before this, every project switch re-latched `done` and re-lit the blue attention dot the user had already acknowledged, so a finished chat pane could never settle back to "ready".
- **A settled `done` is a notification, not a resting state.** Engaging the pane again — clicking it (the same gesture that clears the unread dot via `selectSession`) or typing the next prompt into the composer — acknowledges the finished turn (`acknowledgeCompletedTurn`) and releases the pill `done` -> `idle` ("ready"). `waiting`/`failed` stay put: they still need an answer / carry the error until the next turn starts.

## Shared-folder chip

Every agent pane (plain shells excluded) whose folder overlaps another agent pane's — same folder or nested, across Multi and all workspaces — shows a neutral `×N here` chip in its header (`pane-cwd-conflict-chip`, computed by `frontend/cwdConflicts.ts` in `App.tsx` and passed to all three pane components). Neutral by design: side-by-side agents on one project is normal use, not a mistake. The chip tints amber as `×N active` only while this pane AND an overlapping peer are both mid-turn (`running`/`starting`) — the real collision window, since the harnesses' stale-write checks don't coordinate across processes. The tooltip names the other panes and their scope.

## Entry Points

- `index.html` loads `/frontend/main.tsx`.
- `tsconfig.json` includes `frontend/` for strict renderer typechecking.
- `npm run dev:frontend` starts the Vite renderer on `127.0.0.1:5173`.
